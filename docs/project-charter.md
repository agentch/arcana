# Arcana 项目章程

## 1. 项目目标

用一个开发周期交付可公开访问的 Web MVP，验证用户是否愿意完成抽牌、阅读解读并保存或分享结果；验证通过后，将同一业务内核迁移到微信小程序。

## 2. 用户与场景

### 目标用户

- 对塔罗感兴趣但不熟悉牌义的新手
- 希望通过抽牌进行情绪整理和自我反思的轻度用户
- 偏好仪式感、视觉体验和可分享内容的年轻用户

### 核心场景

1. 用户带着一个问题进入应用。
2. 选择适合问题的牌阵。
3. 完成洗牌、抽牌和翻牌。
4. 阅读牌义与组合解读。
5. 保存记录或生成分享卡片。

## 3. 成功标准

MVP 上线后重点观察：

- 抽牌流程完成率
- 从首页到结果页的中途流失点
- 结果页平均停留时长
- 保存率与分享发起率
- 7 日内再次使用率
- 页面错误率与 Web 核心性能指标

立项阶段不预设虚假的数值目标。首轮真实数据形成基线后，再为第二轮迭代设置量化目标。

## 4. MVP 范围

### 必须有

- 首页与占卜入口
- 今日一牌与单牌占卜
- 时间流、圣三角
- 恋爱关系五牌阵、二选一牌阵
- 78 张牌的基础数据、正逆位牌义
- 洗牌、抽牌、翻牌、结果解读
- 最近记录的本地保存与删除
- 分享卡片
- 基础埋点、异常监控、免责声明
- 移动端优先的响应式 Web 页面

### 应该有

- 问题输入与问题分类
- 无障碍基础支持
- 静音开关和动效降级
- 首次使用引导

### 暂不纳入

- 用户账号与云同步
- 付费解读、订阅与订单
- 社区、评论、私信
- 真人塔罗师撮合
- AI 自由对话
- 多套牌面与用户自定义牌阵

## 5. 内容与合规原则

- 明确说明产品用于娱乐与自我探索。
- 对健康、法律、投资等高风险问题不提供确定性建议。
- 不使用恐吓、宿命论或诱导付费的表达。
- 若收集问题文本、行为数据或设备标识，先取得清晰同意，并提供删除机制。
- 使用第三方牌面前确认版权；无明确授权时使用自有或公版素材。
- 小程序发布前单独复核微信平台规则、隐私指引和类目要求。

## 6. 约束与假设

- 首版以中文、移动端竖屏为主。
- Web 与小程序共享业务逻辑、牌库和大部分 UI，不追求所有平台逐像素一致。
- 首版牌义为审核过的结构化固定内容，AI 解读作为后续可插拔能力。
- 账号与后端不是验证核心价值的前置条件。
- Git 仓库和工程代号使用 `arcana`。
- MVP 采用标准 78 张塔罗体系；牌面优先评估经典 Rider–Waite–Smith 原始版本，使用前必须逐项核验素材来源和授权。
- 数据模型从第一天支持多个牌组，但 MVP 界面只提供一套。

## 7. 主要风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 牌面版权不清晰 | 无法发布或被投诉 | 开发前锁定公版或原创素材授权 |
| Web 特效无法直接迁移 | 小程序返工 | 只在平台适配层使用 DOM、Canvas 等能力 |
| 固定解读缺少个性 | 留存偏低 | 先验证流程，再灰度组合模板或 AI |
| AI 输出绝对化或越界 | 用户风险与平台风险 | 服务端提示词、内容审核、免责声明和降级模板 |
| 随机算法被质疑 | 信任受损 | 使用可测试的洗牌算法，并明确娱乐属性 |
| 过早引入后端 | 延误验证 | MVP 优先本地数据与本地记录 |

## 8. 立项完成条件

- MVP 范围得到确认
- React + Taro 技术基线得到确认
- 牌面版权方案得到确认
- 核心流程低保真原型通过评审
- P0 内容清单与验收标准可执行
