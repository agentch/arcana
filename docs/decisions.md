# Arcana 决策记录

## D-001：仓库名称

- 日期：2026-07-20
- 状态：已决定
- 决策：Git 仓库和工程代号使用 `arcana`。
- 原因：名称与大阿卡纳体系相关，简短且保留后续产品扩展空间。

## D-002：交付顺序

- 日期：2026-07-20
- 状态：已决定
- 决策：先发布移动端优先的 Web，再迁移微信小程序。
- 原因：Web 验证和迭代成本更低，同时需要从第一天隔离平台能力。

## D-003：跨端技术栈

- 日期：2026-07-20
- 状态：已决定
- 决策：使用 React + TypeScript + Taro。
- 原因：符合已有 React 技术背景，并能覆盖 H5 与微信小程序。

## D-004：首版牌组

- 日期：2026-07-20
- 状态：部分决定
- 决策：MVP 使用标准 78 张塔罗体系，优先评估经典 Rider–Waite–Smith 原始版本。
- 未决事项：必须确定每张正式素材的来源、版本和授权；未核验前只能作为原型候选。

## D-005：多牌组能力

- 日期：2026-07-20
- 状态：已决定
- 决策：数据模型支持多个牌组，但 MVP 只展示一套。
- 原因：为原创东方牌组预留空间，又不增加首版界面复杂度。

## D-006：MVP 后端

- 日期：2026-07-20
- 状态：暂定
- 决策：MVP 核心流程不依赖后端，牌库随应用发布，记录保存在本地。
- 复核条件：需要账号、云同步、AI 解读或远程运营配置时重新评估。

## D-007：AI 解读

- 日期：2026-07-20
- 状态：待验证
- 决策：固定牌义和组合模板是必备降级能力；单次 AI 解读为可插拔选项；连续对话不进入 MVP。
- 复核条件：原型完成并明确 AI 能解决的具体用户问题后。

## D-008：产品边界

- 日期：2026-07-20
- 状态：已决定
- 决策：产品定位为娱乐和自我探索工具，不提供医疗、法律、财务或心理咨询结论，不做绝对预测。

## D-009：首版原型视觉方向

- 日期：2026-07-20
- 状态：待评审
- 决策：首版使用午夜靛蓝、旧金色和纸张暖白，以月相、轨道和星点作为视觉母题。
- 原因：建立神秘与仪式感，同时避免恐怖、廉价霓虹和过度神秘学符号。
- 复核条件：用户完成首版原型评审后确认或调整。

## D-010：原型与正式工程分离

- 日期：2026-07-20
- 状态：已决定
- 决策：P1 可点击原型放在 `prototype/`；评审完成后再初始化正式 Taro 工程。
- 原因：原型用于快速确认体验，不将临时实现直接演化为跨端生产代码。

## D-011：首批牌阵优先级

- 日期：2026-07-20
- 状态：已决定
- 决策：按单牌、时间流、圣三角、恋爱关系五牌阵、二选一牌阵的顺序开发。
- 原因：从单牌核心闭环逐步增加多牌布局、组合语义、专题引导和分支对比，技术与内容复杂度递进。

## D-012：时间流与圣三角分离

- 日期：2026-07-20
- 状态：待原型确认
- 决策：时间流使用“过去 / 现在 / 未来”；圣三角首版使用“现状 / 挑战 / 建议”。
- 原因：避免两个三牌阵只有名称不同，使用户能够根据问题类型明确选择。

## D-013：二选一不替用户决策

- 日期：2026-07-20
- 状态：已决定
- 决策：二选一牌阵对比两条路径的过程与倾向，不输出“必须选 A”或“必须选 B”的确定结论。
- 原因：保持自我探索定位，降低宿命论和高风险决策误导。

## D-014：根目录唯一 Git 仓库

- 日期：2026-07-20
- 状态：已决定
- 决策：Arcana 根目录作为唯一 Git 仓库，`prototype/` 不保留嵌套 `.git`。
- 原因：项目文档、原型、后续正式工程、牌义 JSON 和素材配置需要统一版本管理；嵌套仓库会导致根仓库只记录 gitlink，无法正常跟踪原型源码。

## D-015：牌义、素材和牌阵三层分离

- 日期：2026-07-20
- 状态：已决定并在原型落实
- 决策：每张牌的稳定语义存入结构化 JSON；牌阵从独立 JSON 加载；牌组清单只维护 `cardId → 素材` 映射。
- 约束：牌义数据不得包含图片路径；页面不得写死牌阵位置；切换牌组不得复制牌义。
- 版本：历史记录同时保存牌义内容版本、牌组版本和牌阵版本。

## D-016：进度汇报必须包含下一步

- 日期：2026-07-20
- 状态：已决定
- 决策：每次阶段性汇报和最终交付都列出“下一步打算”，包括执行顺序、目标和需要用户确认的事项。
- 原因：让项目推进方向始终可见，方便及时调整优先级。

## D-017：数据校验作为构建前置门槛

- 日期：2026-07-20
- 状态：已决定并落实
- 决策：牌义、牌组和牌阵均使用 JSON Schema；完整测试必须先执行 `data:validate`，再进入应用构建。
- 额外检查：card ID 唯一且排序、牌组映射完整、牌阵位置唯一且连续、大阿卡纳编号连续。
- 原因：内容规模增长后，尽早阻止结构错误、漏图和错误关联进入页面。

## D-018：问题引导采用分类与选项两级结构

- 日期：2026-07-20
- 状态：已决定并落实
- 决策：问题引导分为感情、职场、家庭、心情；每类提供4个具体选项，选择后填入开放式问题，用户仍可自由编辑。
- 数据：分类、选项和提示问题存入独立 JSON；历史记录保存分类 ID、选项 ID 和最终问题文本。
- 原因：降低新用户提问门槛，同时保留开放表达和后续主题分析能力。

## D-019：首套正式牌组使用原始 RWS

- 日期：2026-07-20
- 状态：牌组体系已决定，具体扫描来源待核验
- 决策：首套正式牌面使用1909/1910年代原始 Rider–Waite–Smith 完整78张，不使用现代重新上色、高清重绘或商业修复版本。
- 一致性：78张牌必须来自同一来源、同一批次的完整扫描，不混用不同版本。
- 文件：保留未改动原始扫描，并通过可重复脚本生成 Web 优化版本；两者均记录 SHA-256、尺寸、大小和格式。
- 授权：正式入库前逐项核验原始作品、具体版本和扫描文件许可，不因画面年代久远而默认现代扫描可以商用。
- 分层：基础牌义、牌组素材映射和牌背继续独立管理。

## D-020：问题选项推荐牌阵但允许手动切换

- 日期：2026-07-20
- 状态：已决定并落实
- 决策：每个问题选项配置 `recommendedSpreadId`；用户选择选项时自动选中推荐牌阵，但仍可在已开放牌阵中手动切换。
- 当前开放：单牌、时间流。
- 原因：为新用户降低选择成本，同时避免系统替用户强制决定牌阵。

## D-021：时间流使用通用无放回抽牌

- 日期：2026-07-20
- 状态：已决定并落实
- 决策：抽牌数量完全由牌阵位置配置决定；使用 Fisher–Yates 洗牌并无放回抽取，正逆位独立生成。
- 测试：随机源可注入，保证测试可复现；同次时间流不得出现重复牌。
- 交互：过去、现在、未来三张牌逐张翻开，全部翻开后才能进入完整解读。

## D-022：正式牌义采用分层内容模型

- 日期：2026-07-20
- 状态：已决定，待 P4 实施
- 决策：正式牌义按核心、正位、逆位和主题四层组织，不再扩展单个长 `meaning` 字段。
- 字段：正逆位均包含关键词、基础解读、积极力量、潜在风险、1–2条行动建议和1条反思问题；核心层保存摘要、象征和元素。
- 主题：候选主题为感情、职场、财务和个人成长；主题内容只补充语境，不复制基础牌义。与现有“感情、职场、家庭、心情”问题分类的映射在 Schema v2 实施前确认。
- 组合：运行时使用“基础牌义 + 正逆位 + 问题主题 + 牌阵位置”生成单牌解读，多牌阵在各牌摘要之后生成整体组合解读。
- 展示：单牌控制在约300–500字；多牌阵默认展示摘要和最终组合解读，详细单牌内容按需展开。
- 迁移：现有22张大阿卡纳短文案作为草稿迁移输入，不视为正式审核内容。
- 详细规范：见 `docs/card-content-model.md`。

## D-023：正式数据基础优先于扩展牌阵

- 日期：2026-07-20
- 状态：已决定
- 决策：时间流完成后，先建立牌义 Schema v2、结构化解读组合器、78张牌组 Manifest、素材处理管线，并将单牌改为完全配置驱动；通过验收后再开发圣三角。
- 原因：继续基于原型短字段开发更多牌阵会扩大页面耦合，后续迁移正式牌义时产生重复返工。
- 并行项：RWS 来源授权核验与78张牌义编写可并行进行；授权未完成只阻塞正式图片入库，不阻塞数据结构和读取链路开发。
- 提交：Schema、组合器、Manifest 和素材接入分别提交并独立验证。

## D-024：问题分类显式映射到牌义主题

- 日期：2026-07-20
- 状态：已决定并落实
- 决策：牌义主题枚举包含 `love`、`career`、`family`、`mood`、`finance`、`growth`；当前四类问题使用同名映射，财务和个人成长作为后续分类预留。
- 配置：映射保存在 `meaning-topic-map.json`，并通过 Schema 和关联校验保证每个启用的问题分类都有对应主题。
- 原因：家庭和心情具有独立语境，不应被页面强行归入感情或个人成长；同时保留未来扩展空间。

## D-025：组合器统一 v2 与原型牌义输出

- 日期：2026-07-20
- 状态：已决定并落实
- 决策：结果页只消费结构化 `InterpretationView`；组合器优先读取完整 v2 牌义，并将尚未完成编辑的 v1 牌义适配成同一展示模型。
- 组合输入：牌、正逆位、问题主题和牌阵位置；输出包含摘要、详细解读、主题内容、牌位提示、象征、明暗面、建议和反思问题。
- 展示：单牌展示完整分层内容；多牌阵保持摘要和单条建议，避免页面过长。
- 原因：在逐张补齐正式牌义期间保持原型所有22张牌可用，同时保证页面不再直接读取旧牌义字段。
- 退出条件：78张 v2 牌义完成后删除 v1 适配层，不修改结果页组件。

## D-026：正式牌组使用78张固定索引和状态化素材清单

- 日期：2026-07-20
- 状态：已决定并落实
- 决策：标准卡牌索引固定为22张大阿卡纳和四个花色各14张小阿卡纳；正式牌组 Manifest 必须逐项映射全部78个稳定 `cardId`。
- 素材状态：每张牌依次经过 `pending-source`、`source-ready`、`web-ready`、`approved`；状态决定必须存在和校验的文件字段。
- 文件记录：原图与 Web 图分别记录相对路径、原始文件名、媒体类型、宽高、字节数和 SHA-256。
- 转换：Web 图由固定参数生成 WebP，不覆盖、不裁切或重新上色原文件；生成后重新读取元数据和校验值。
- 牌背：独立于78张牌面和牌义管理；来源不明确时使用后续审核的 Arcana 原创牌背。
- 授权门槛：牌组来源和许可未通过审核时，任何牌面都不能标记为 `approved`。

## D-027：RWS 候选来源选用 Commons TaionWC Pam-A 集合

- 日期：2026-07-20
- 状态：技术审计通过，许可待项目方最终接受
- 候选：Wikimedia Commons 的 `Rider-Waite-Smith tarot deck (TaionWC)` 分类。
- 完整性：分类明确为 TaionWC 成套上传的 Pam-A image set，共78个文件。
- 一致性：78个文件均为 JPEG，宽1090–1144、高1919–1920；作者、原作日期和使用条款字段一致。
- 权利元数据：78个文件均标记 Public domain、Public Domain Mark 和 `PD-old-80-expired`。
- 固化：逐文件页面、下载地址、当前版本、SHA-1、尺寸和许可字段写入 `source-audit.json`，并生成不可变 `sourceBatchId`。
- 边界：Wikimedia 不提供权利状态保证；在项目方接受该风险前，许可保持 `pending-review`，不把文件标记为正式批准素材。
