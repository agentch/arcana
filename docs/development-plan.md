# Arcana 开发计划

版本：v1.1
制定日期：2026-07-20
当前阶段：立项与原型

执行状态：P1 首版原型已实现，等待产品评审。

## 1. 交付目标

先交付一个移动端优先、可公开访问的 Web MVP，验证用户能否顺畅完成“提出问题—抽牌—查看解读—保存或分享”的完整流程。Web 验证完成后，使用同一套领域逻辑和内容数据迁移到微信小程序。

## 2. 开发原则

1. 每个阶段必须有可演示成果和明确验收标准。
2. 先跑通单牌闭环，再按“时间流、圣三角、恋爱关系五牌阵、二选一牌阵”的顺序扩展。
3. 牌义、随机算法和占卜记录属于领域层，不依赖浏览器或微信 API。
4. AI、登录、支付和后台不得阻塞 Web MVP。
5. 任何第三方牌面进入正式版本前，必须记录来源、作者、版本和授权证据。
6. 每完成一个里程碑，同步更新 `progress.md`。
7. 每次进度汇报和任务交付都必须列出“下一步打算”，说明后续工作顺序、目标与待确认项。

## 4. P1：移动端可点击原型

### 目标

在不接入真实后端和完整牌库的情况下，确认首版产品结构、交互节奏和视觉语言。

### 页面

- 首页
- 占卜方式选择
- 问题输入
- 洗牌与抽牌
- 翻牌
- 解读结果
- 历史记录

### 临时内容

- 愚人
- 恋人
- 星星
- 单牌阵
- 时间流三牌阵

### 验收标准

- 以 390 × 844 移动端视口完成主流程。
- 用户无需说明即可找到“开始占卜”入口。
- 从首页到结果页不超过 6 个主要操作。
- 动画跳过或失败时仍能完成抽牌。
- 正位和逆位在视觉与文案上都能明确区分。
- 结果页同时呈现牌名、牌位、关键词、解释和行动建议。
- 原型评审后形成页面清单与交互修改记录。

### 2026-07-20 实施结果

- 原型位置：`prototype/`
- 已完成：首页、问题输入、洗牌、抽牌、翻牌、结果和历史记录。
- 已完成：完整22张大阿卡纳结构化样例内容。
- 已完成：正逆位、跳过动效、本地保存、减少动态效果支持。
- 已完成：Arcana 专属社交分享预览图。
- 已完成：牌义、牌组素材、牌阵配置拆分为独立 JSON，并增加关联完整性测试。
- 已完成：三类正式 JSON Schema、独立数据校验命令和完整22张大阿卡纳结构化内容。
- 已完成：感情、职场、家庭、心情四类问题引导，每类提供4个可选主题和可编辑问题。
- 已完成：16个问题选项的推荐牌阵映射，以及单牌、时间流手动切换。
- 已完成：时间流三张无重复抽取、逐张翻牌、分牌位解读和版本化历史记录。
- 已验证：生产构建和服务端渲染测试通过。
- 已发布：私有评审版本 `https://arcana-tarot-prototype.lostme037.chatgpt.site`。
- 待完成：用户视觉与流程评审；根据反馈修订后关闭 P1。
- 暂未纳入：正式牌面素材、完整 78 张牌库、扩展牌阵、真实分享能力。

- 校验牌义不包含图片字段、牌组完整映射所有 card ID、牌阵位置连续且唯一。
- 增加基础构建检查。
- 确认 H5 与微信小程序空壳均能构建。

### 验收标准

- 新环境按照 README 可以完成安装和启动。
- H5 开发模式可访问。
- H5 正式构建成功。
- 微信小程序构建产物可被开发者工具打开。
- 随机算法示例测试通过。
- `npm run data:validate` 可以独立校验内容，不依赖页面构建。
- Schema v2、组合器和78张 Manifest 骨架均有自动测试或校验。
- 业务代码没有直接散落浏览器或微信平台 API。

## 6. P3：单牌闭环

### 用户故事

作为第一次使用塔罗的用户，我希望输入一个问题并抽取一张牌，看到容易理解且不过度断言的解读。

### 工作项

- 首页和新手说明
- 问题输入与问题示例
- 单牌洗牌、选牌和翻牌
- Fisher–Yates 洗牌算法
- 正逆位生成
- 愚人完整分层样例和大阿卡纳迁移草稿
- 单牌完全读取配置和结构化展示模型
- 结果页
- 本地历史记录
- 免责声明
- 基础事件记录

### 核心测试

- 同次占卜不能重复抽牌。
- 固定随机种子产生可复现结果。
- 正逆位配置生效。
- 历史记录包含牌组版本和内容版本。
- 本地数据异常时应用可以降级启动。
- 页面刷新后已保存记录仍能恢复。
- 单牌业务逻辑不包含牌义、图片路径或牌阵位置硬编码。

### 验收标准

- 主路径可在移动端浏览器完整执行。
- 无需账号、网络 API 或 AI 即可完成占卜。
- 结果可保存、查看和删除。
- 高风险问题显示明确边界说明。

## 7. P4：完整 Web MVP

P4 按“完整内容 → 扩展牌阵”的顺序执行。进入 P4 前，必须先在 P2/P3 完成正式数据读取链路；圣三角不得先于该链路开发。

### P4 进入门槛：正式数据基础

| 顺序 | 阶段 | 工作项 | 交付物 | 预计工作量 |
| --- | --- | --- | --- | --- |
| 1 | P2 | 建立牌义 Schema v2 | 分层 JSON Schema、样例、迁移与校验脚本 | 1–2 天 |
| 2 | P2 | 实现结构化解读组合器 | 方位、主题、牌位的统一输出模型与测试 | 1–2 天 |
| 3 | P2 | 升级牌组图片映射和素材管线 | 78张 Manifest 骨架、牌背、来源、授权、校验值和处理命令 | 1–2 天 |
| 4 | P3 | 单牌完全配置化 | 页面只消费组合后的展示模型 | 1–2 天 |

以上门槛完成后才开始圣三角。完整78张牌义和正式RWS图片可以在统一接口下继续补充，不再要求修改页面业务逻辑。

### P4-A：完整内容

- 核验同一来源、同一批次的原始 RWS 78张扫描及其授权。
- 来源确认后接入78张原图，保留原文件并生成 Web 优化版本。
- 按“大阿卡纳 → 权杖 → 圣杯 → 宝剑 → 钱币”补齐并审核正式牌义。
- 完整牌义编写与两轮审核预计7–14天；素材来源确认后的处理与接入预计1–2天。
- 授权核验未完成时，正式牌面保持阻塞，但不能阻塞 Schema、组合器、Manifest 和页面配置化。

### P4-B：扩展牌阵

- 圣三角。
- 恋爱关系五牌阵。
- 二选一牌阵。
- 每个牌阵复用 P2/P3 的配置读取与解读组合器，不增加牌阵专属牌义副本。

### 工作项

- 确认 P2/P3 正式数据基础已经通过进入门槛。
- 完成 P4-A 完整78张牌义和原始 RWS 牌面。
- 完成 P4-B 三种扩展牌阵。
- 增加今日一牌。
- 增加时间流。
- 增加组合解读模板。
- 完成分享卡片。
- 增加声音和减少动态效果设置。
- 增加空状态、错误状态和首次引导。
- 优化图片加载、缓存与移动端性能。

### 内容验收

- 每张牌拥有稳定 ID、中英文名、编号、类别、牌组和版本。
- 小阿卡纳包含花色与宫廷牌分类。
- 每张牌包含核心摘要、3–5 个牌面象征、正逆位明暗面、1–2 条行动建议和 1 条反思问题。
- 每张牌覆盖最终确认的统一主题枚举。
- 正逆位内容不是简单的正反义替换。
- 单牌总阅读量控制在约300–500字，多牌阵默认展示摘要并将组合解读置于最后。
- 文案不使用恐吓、保证结果或诱导付费表达。
- 所有图片都能追溯来源和授权状态。

### 产品验收

- 单牌、今日一牌、时间流、圣三角、恋爱关系五牌阵和二选一牌阵全部可完成。
- 分享失败不会影响查看和保存结果。
- 弱网下首屏和核心操作仍有反馈。
- 主流移动浏览器无阻断性问题。

### 牌阵实施顺序

1. 单牌：建立所有牌阵共用的抽牌、正逆位、结果和历史记录能力。
2. 时间流：已验证多牌抽取、去重、逐张翻牌和横向三牌布局。
3. 正式数据基础：完成 Schema v2、组合器、牌组 Manifest、素材处理流程和单牌完全配置化。
4. 圣三角：验证同样张数下不同牌位语义与组合解读。
5. 恋爱关系五牌阵：验证专题问题引导和五牌响应式布局。
6. 二选一牌阵：验证分支对比解读，确保不替用户直接做决定。

每完成一种牌阵均独立验收和提交，不等待所有牌阵一次性完成。

## 8. P5：Web 发布

### 发布前检查

- 隐私说明、免责声明和素材归属完整。
- 页面标题、分享信息和基础 SEO 完整。
- 错误监控和必要事件统计已配置。
- 不上传问题文本等敏感内容，除非用户已明确同意。
- 完成移动端性能、无障碍和兼容性检查。
- 生产环境没有模型密钥或其他服务密钥。
- 建立回滚版本。

### 上线后观察

- 首页到开始占卜的转化
- 开始到完成抽牌的转化
- 结果页到保存或分享的转化
- 主流程错误率
- 7 日内回访情况
- 用户对解读准确性、清晰度和情绪安全感的反馈

## 9. P6：AI 解读决策门

AI 不自动进入 MVP。满足以下条件后再开发：

- 固定牌义和组合模板已完成。
- 主流程已有真实反馈。
- 明确 AI 要改善的指标或用户问题。
- 已确定服务端代理、成本上限、超时策略和内容审核。
- AI 失败时可立即回退到固定模板。

若启用，第一版仅生成一次结构化解读：

1. 当前处境
2. 牌面提示
3. 可执行建议
4. 边界提醒

不做无限追问，不允许客户端直接保存模型密钥。

## 14. 完成定义

一个任务只有同时满足以下条件才可标记完成：

- 功能符合对应验收标准。
- 必要测试通过。
- 移动端主路径已手动验证。
- 异常和空状态已处理。
- 相关文档与进度记录已更新。
- 没有引入未经记录的素材或服务依赖。
