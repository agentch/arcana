## 阶段 1：工程基础

- 初始化 Taro + React + TypeScript 工程
- 配置代码规范、单元测试和构建检查
- 建立设计令牌、基础布局与跨端适配接口
- 定义牌库 schema，并加入内容校验脚本
- 建立 Web 预览发布流程

退出条件：Web 和微信小程序空壳均能构建，核心测试在持续集成中通过。

## 阶段 2：Web MVP

### P0

- 首页与模式选择
- 单牌、时间流、圣三角领域逻辑
- 恋爱关系五牌阵、二选一牌阵领域逻辑
- 洗牌 / 抽牌 / 翻牌交互
- 结果页与固定牌义
- 本地历史记录
- 分享卡片
- 免责声明与隐私说明
- 基础埋点、异常监控和性能检查

### P1

- 今日一牌
- 首次引导
- 动效与声音设置
- 无障碍与弱网体验优化

退出条件：核心流程在主流移动浏览器通过验收并可公开访问。

## 阶段 3：数据验证

- 收集首轮漏斗与留存基线
- 访谈或可用性测试
- 修复主要流失点
- 决定是否引入账号、云同步、AI 解读或新牌阵
- 若引入 AI，先实现单次结构化解读、内容审核与固定模板降级

退出条件：形成小程序迁移决策与第二轮量化目标。

## 阶段 4：微信小程序

- 增加微信应用配置
- 适配导航、安全区、存储、分享和隐私授权
- 处理图片包体积与分包
- 小程序真机兼容性测试
- 完成平台合规材料与提审

退出条件：审核通过并完成灰度发布。

## 首个开发迭代建议

迭代目标：从首页进入单牌占卜，完成抽牌并看到一个可复现的结果。

1. 初始化工程与质量工具。
2. 定义牌库 schema，先录入 3 张样例牌。
3. 实现随机、正逆位和单牌牌阵领域逻辑及单测。
4. 完成首页、抽牌页、结果页最小闭环。
5. 加入本地记录适配接口。
6. 在 Web 移动端完成一次端到端验收。

## MVP 验收主路径

```text
进入首页
  → 选择占卜方式
  → 输入或跳过问题
  → 洗牌
  → 抽牌
  → 翻牌
  → 阅读解读
  → 保存或分享
```

任何动画、音效或登录失败，都不应阻断这条主路径。
